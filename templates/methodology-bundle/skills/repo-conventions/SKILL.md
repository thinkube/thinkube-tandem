# Repository conventions

Hand-edit this file after installing the bundle. The bundle ships with the defaults below; replace each section with what your project actually uses. Other bundle skills (`/pair-next`, `/slice`, `/spec-prepare`) load this file on demand to ground their suggestions. The `verifier` subagent reads the **verification recipe** below.

## Branches

Default: one branch per slice, named `slice/SP-{n}_SL-{m}`.

```
slice/SP-3_SL-4
slice/SP-3_SL-5
```

Branches go to `main` via PR. Long-lived feature branches are an exception, not the default.

## Pull requests

- Title: `<slice title>` (matches the slice's one-line description).
- Body opens with a one-line summary + the slice handle it closes (`Closes SP-3_SL-4`).
- Description includes:
  - What changed (1–2 sentences).
  - Files touched (auto-generated by `gh pr create` is fine).
  - How to verify (the verification recipe below; any manual steps).
- Reviewers: per the team's CODEOWNERS or the user's preference.

## Commits

Conventional Commits-ish but flexible:

```
<type>(<scope>): <one-line summary>

<optional paragraph>

Refs SP-3_SL-4
```

Types in active use: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `build`. Scope is optional.

## Verification recipe (the `verifier` reads this)

The `verifier` subagent runs the commands below to gate a slice's move to **Done**. There is **no separate test-suite assumption** — the recipe is the source of truth. The general rule, and the recipe **for this repo**, is:

- Host typecheck: `tsc -p ./ --noEmit`
- Webview build: `npm run compile:webview`
- Tests: `npm test`

All three must pass (exit 0) for a green. Replace these for a different project if its toolchain differs; where a project has no test suite, the typecheck + build pair stands in for "one green."

## Things to never do

- `git push --force` to `main`.
- `rm -rf` against `.git/`, `node_modules/`, or anywhere outside the workspace root.
- Skip `pre-commit` / `pre-push` hooks via `--no-verify`. If a hook fails, fix the underlying issue.
- Commit secrets. The `ThinkubeStore.writeFile` secret-scan catches the obvious patterns, but eyes on the diff before pushing remains the human's responsibility.
