# Repository conventions

Hand-edit this file after installing the bundle. The bundle ships with the defaults below; replace each section with what your project actually uses. Other bundle skills (`/pair-next`, `/tasks-decompose`, `/spec-prepare`) load this file on demand to ground their suggestions.

## Branches

Default: one branch per Task, named `task/<issue-number>-<short-slug>`.

```
task/142-wire-oauth-callback
task/143-redis-session-store
```

Branches go to `main` via PR. Long-lived feature branches are an exception, not the default.

## Pull requests

- Title: `<Task title>` (matches the issue title).
- Body opens with a one-line summary + a link to the Task issue (`Closes #142`).
- Description includes:
  - What changed (1–2 sentences).
  - Files touched (auto-generated by `gh pr create` is fine).
  - How to verify (the test command that should pass; any manual steps).
- Reviewers: per the team's CODEOWNERS or the user's preference.

## Commits

Conventional Commits-ish but flexible:

```
<type>(<scope>): <one-line summary>

<optional paragraph>

Refs #<issue-number>
```

Types in active use: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `build`. Scope is optional.

If a project uses multiple test runners (unit + integration + e2e), list the command that covers the **first two** at minimum; e2e is run by the user explicitly.

## Things to never do

- `git push --force` to `main`.
- `rm -rf` against `.git/`, `node_modules/`, or anywhere outside the workspace root.
- Skip `pre-commit` / `pre-push` hooks via `--no-verify`. If a hook fails, fix the underlying issue.
- Commit secrets. The `ThinkubeStore.writeFile` secret-scan catches the obvious patterns, but eyes on the diff before pushing remains the human's responsibility.
