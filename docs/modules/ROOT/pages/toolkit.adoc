= Skills, subagents, and tools
:page-type: reference
:page-docref: tk7d

The Tandem toolkit is the set of moving parts the pair drives: the *skills* that author artifacts, the *subagents* that research and check work, the *board actions* you take in the panel, the *MCP tools* the AI calls to read and write the board, the *hooks* that run automatically, and the per-repo *verification recipe*. This page catalogues each one and marks who runs it — you or the AI.

== Who runs what

* *Skills* are invoked in chat, by you or by the AI, and drive the board through MCP tools.
* *Subagents* are delegated to by skills; they research or verify and never touch the board.
* *Board actions* are buttons in the thinking-space panel. They are *yours* — the AI never presses them. It stops at a bless point, a gate refusal, or a failed precondition and hands back to you.
* *MCP tools* are the AI's programmatic surface onto the board and thinking space.
* *Hooks* run on their own trigger (turn end, session start) with no one invoking them.

== User-invocable skills

Each skill routes writes through a board-aware MCP tool so files land in the central thinking-space sidecar, never on the session's working repo by a raw write.

[cols="1,3,2",options="header"]
|===
| Skill | Mission | Output

| `/tep`
| Author a Tandem Enhancement Proposal — the _why_ above the work, recorded before the specs that reference it.
| A TEP file in the thinking space (via `write_tep`).

| `/spec-prepare`
| Fill a Spec's body to the Tandem shape — acceptance criteria, constraints, design, file plan — and certify every criterion verifiable before the → Ready gate.
| A Spec document in the thinking space (via `write_spec`).

| `/slice`
| Decompose a shaped Spec into coherent end-to-end slices, each verified-and-committed as a single "done."
| One slice file per slice (via `create_slice`).

| `/thinking-space`
| Snapshot the current board's Ready / Doing / Done columns, read from slice frontmatter, and point you at the interactive panel.
| A read-only table in chat; no file written.

| `/retro`
| Append a structured retro note under one lens — kept, changed, learned, or blocked.
| A dated entry in today's retros file (via `write_retro_note`).
|===

`/attend` is also a skill, but you reach it through the *Attend* or *Request changes* board actions (see below) rather than invoking it cold; it opens a primed fix session on a requires-attention slice or a rejected-Spec rework. The session opens in the *canonical repo*, so it survives the worktree's retirement at Accept; the prompt names the Spec's worktree, where the fix is applied and committed to the spec branch (`fix(attend): …`). The failure evidence — the judged fault and the failing output — arrives in the prompt verbatim; grading independence lives in the closing gate's assessor and judge, which re-grade the committed state on the next run.

== Reference skills

These are loaded on demand by the skills above to ground themselves. They are not directly invocable (`disable-model-invocation`).

[cols="1,3",options="header"]
|===
| Skill | Role

| `methodology-context`
| The shared Tandem vocabulary, hierarchy, and workflow. Other bundle skills load it when they need the common terms.

| `repo-conventions`
| Per-project branch naming, PR template, commit format, and the *verification recipe*. Hand-edited per repo after the bundle is installed; the `verifier` subagent reads its recipe.
|===

== Subagents

All three are read-only and return findings to the main conversation; none writes code or the board.

[cols="1,2,1,2",options="header"]
|===
| Subagent | Role | Tools | Refuses

| `explorer`
| Read-only codebase research — file structure, existing patterns, prior implementations — returned with `file:line` references so the main context stays lean.
| Read, Grep, Glob, Bash
| Any write; anything outside the workspace; "while you're at it" fixes.

| `reviewer`
| Adversarial diff review against the parent Spec's acceptance criteria; returns blockers, risks, nits, and per-AC coverage. Runs inside the single Done gate.
| Read, Grep, Glob, Bash
| Writing code; running builds or tests; a blanket "looks good" with no specifics.

| `verifier`
| Runs the repo's verification recipe and returns a pass/fail with evidence. Gates a slice's move to Done.
| Read, Grep, Glob, Bash
| Writing code; editing tests to pass; running checks outside the recipe.
|===

== Board actions you run

These are the human-only buttons in the thinking-space panel. The AI can open the surfaces that carry them — for example `open_review` mounts the review panel — but it cannot press them.

[cols="1,3",options="header"]
|===
| Action | What it does

| *Approve spec*
| Grants the content-bound approval token that unlocks slicing and the → Ready gate. Only you can grant it; the AI opens the panel with `open_review` but never approves.

| *Orchestrate*
| Starts the run that schedules and executes a Spec's work-unit DAG to its closing gate.

| *Start Spec in Worktree*
| Opens the Spec's dedicated git worktree pair session.

| *Accept & merge*
| The delivered-state exit: accept the Spec and squash-merge its one PR to `main` — one verified commit per Spec, the working trail kept on the PR — gated on every acceptance criterion being checked.

| *Request changes*
| The delivered-state exit's counterpart: open a primed `/attend` session (in the canonical repo) to steer the delivered change back to the intent.

| *Attend*
| The stalled-state exit: open a primed `/attend` session on the requires-attention slice(s). The session opens in the canonical repo; the fix is applied in the Spec's worktree and committed to the spec branch. You only reach it after the loop's automated remedies — judge-routed rework, gate self-heal, the circuit breaker, and the auto-attend fixer — have all run.

| *Re-run*
| The stalled-state exit: after resolving the requires-attention slice(s), re-run the orchestrator. A returning slice with every unit checkpointed is graded as-is; re-dispatch schedules only un-checkpointed units.

| *Reload Window*
| VS Code's window reload. Recovers orphaned session tabs after the worktree layout changes (for example a worktree removed at acceptance).
|===

The exit states behind these buttons are described in xref:lifecycle.adoc[The Spec lifecycle]; the gates they pass through are in xref:gates.adoc[Gates and verification].

== Kanban MCP tools

The AI's surface onto the board and thinking space, grouped by purpose. Every tool named here is registered by the kanban MCP server.

Every tool that names a Spec or a slice accepts *one shared ref grammar*. Spec refs: `1/4`, `TEP-1/SP-4`, `SP-1/4`, `TEP-1_SP-4`, or a bare `SP-4` / `4` — the bare forms are looked up against the thinking space's TEPs and refused when ambiguous or unknown. Slice refs: `TEP-1_SP-4_SL-1`, `SP-4_SL-1`, or `1/4/1`. An unrecognized ref fails loudly, stating the grammar; ids are strictly numeric and sequential, minted per thinking space and org (the per-maintainer org segment is what keeps numbers collision-free).

=== Authoring

[cols="1,3",options="header"]
|===
| Tool | Purpose

| `write_tep` | Create or update a TEP file in the thinking space.
| `write_spec` | Create or update a Spec document, preserving existing frontmatter such as `accepted:`. Refuses to sign an `ac_verifications` command whose leading token cannot resolve in the working repo — repo-local tools go through `npx`, `uv run`, or `poetry run`.
| `create_slice` | Create a slice under a Spec in canonical shape; the server allocates the slice number and enforces the → Ready gate. Refuses a footprint path that does not exist in the Spec's working repo, with a did-you-mean; genuinely-new files are declared in `creates:` (held-out test-unit footprints are exempt).
| `update_slice` | Update a slice in place — replace its body, or re-cut its footprint, `satisfies`, or work units — keeping its number. Enforces the same footprint existence gate as `create_slice`.
| `patch_spec_section` | Replace exactly one named section of a Spec's body, leaving every other section byte-identical.
|===

== Bundle hooks

Hooks run automatically on their trigger. Two are armed in the bundle; the third ships but is deliberately not wired.

[cols="1,1,1,3",options="header"]
|===
| Hook | Trigger | Armed | What it does

| board autosave
| Stop (turn end)
| Yes
| Stages the whole board working tree, commits, rebases, and pushes the sidecar, so board state is never left only on local disk. Fail-open — it never wedges the turn.

| stale-cwd warning
| SessionStart (resume)
| Yes
| Warns when a *resumed* session's original working directory is gone (for example a worktree removed at acceptance) and recommends reopening a fresh session from the board. Warn-only. A legacy safety net for sessions opened before attend sessions moved to the canonical repo — new attend sessions do not strand.

| ownership edit guard
| PreToolUse (Edit / Write)
| No
| Would refuse an edit to a file the active slice does not own. Shipped in the bundle but intentionally not armed — see Discoveries.
|===

== The per-repo verification recipe

`.tandem/conventions.json` is this repo's verification recipe — the language-specific half that lets the same methodology run against a TypeScript extension, a Python package, a Rust crate, or an Ansible role. The `verifier` subagent and the closing gate read it.

[cols="1,3",options="header"]
|===
| Field | What it declares

| `selfVerify`
| The canonical non-mutating build-and-test command a code worker runs to self-check its edits before the gate.
| `testExample`
| A repo-relative path to a canonical example test, injected into test workers so they follow the repo's fixture-and-assertion idiom.
| `acceptanceProbe`
| How a runnable acceptance criterion is graded by an independently authored probe — `sourcePath` (where the probe is written), an optional `prepare` build step, and the per-criterion `run` command.
|===

The slice and Spec artifacts these tools and gates operate on are described in xref:artifacts.adoc[Artifact shapes], and the hierarchy they nest in is in xref:hierarchy.adoc[TEPs, Specs, and Slices].

== Discoveries

* *The ownership edit guard is shipped but not enforcing.* `ownership-guard.mjs` exists in the bundle, but the hook manifest deliberately does not arm it: it still performs a blocking stdin read of the kind that has hung sessions, and it is a no-op until the orchestrator sets an active-slice signal. Both must be resolved before it can protect parallel-slice footprints. Until then the guard is documented but does not run.
* *The MCP tool descriptions and the skills disagree on the Spec path shape.* Several tool descriptions pin `specs/SP-\{n}/spec.md`, while the skills and methodology use the nested `teps/TEP-\{t}/SP-\{n}/spec.md` tree. The surface documents two path shapes for the same file.
