= Thinking Spaces
:description: How thinking spaces work in Thinkube Tandem — the living checklists that produce TEPs.

A *thinking space* is a named, durable surface inside a repo or project where you accumulate intent as checklists of discrete items.
Unlike the old scratchpad, a thinking space outlives any single proposal: you freeze a subset of items into a TEP, the space stays open, and the next TEP reuses all the research and context already gathered.

== The tree

Thinking spaces appear as named nodes in the *Thinking Spaces* sidebar view.
Each repo or project node expands to list its named thinking-space documents, with a *New thinking space...* node at the end.

[source]
----
Thinking Spaces
+-- my-repo                      (repo node -- click to expand)
|   +-- feature-x                (thinking-space document)
|   +-- api-redesign             (thinking-space document)
|   +-- New thinking space...    (creates a new document)
+-- Platform / my-project        (project node -- click to expand)
    +-- q3-intent
    +-- New thinking space...
----

Click a *thinking-space document* node to open its panel.
Click *New thinking space...* to be prompted for a name, after which the panel opens on the freshly seeded document.

A fresh document seeds exactly one empty checklist per section kind: _goal_, _constraints_, _elements_, _gap_, _criteria_, and _verification_.

== Sections and items (checklists)

Every section except the intent editor shows its content as a list of discrete items.
Each item carries:

* A *checkbox* -- only you (the human) can check it; a worker proposal always lands unchecked.
* A *modality marker* -- *mandatory* or *optional*, shown beside the item.
* *Evaluation facets* -- *complexity* and *risk*, each rated 1--3 when relevant.
* *Origin* -- where the item came from: human, gap-filler, integrator, or research.
* *State* -- active, shipped, deferred, or dropped (see below).
* *Evidence chips* -- research findings attached to the item, each carrying method, date, and a dossier reference.

You can add an item to any section at any time.
Items added by hand (direct manipulation or the command field) are born checked; worker-proposed items are born unchecked and await your decision.

== The intent editor

The panel contains exactly one intent editor at the top.
Write your rough intent there -- a few lines is enough to seed the first prefill round.
The reframe worker rewrites the intent statement from your *checked* items only; the original draft is the seed, not the output.

== Only you settle

The reducer enforces this mechanically: any action that would check an item and whose actor is not `human` is *rejected* -- the model reference stays unchanged and the delta is logged as rejected.
Workers can *propose* items, *propose edits* to checked items, and attach *evidence*, but they can never set the checkbox.
The command field counts as human: typing "accept all constraints" checks those items on your behalf.

== Worker rounds and their visibility

Worker rounds are visible while they run.
While a round is in flight:

* The control that started it carries the `disabled` attribute.
* Every section the round targets shows `data-activity="running"`.

When a round completes, sections flip to `data-activity="landed"`.
When a round fails, sections flip to `data-activity="failed"` and a round-error element appears inside the section -- never a silent nothing.

The rounds:

*Prefill* (triggered by clicking *Prefill*)::
Runs the gap-filler worker.
The prompt carries your intent text, every existing item including shipped ones, and each item's evidence dossier references -- the space's full accumulated context travels with every round.
Proposed items land unchecked in their sections with recommended modality and evaluations.

*Reframe* (triggered by clicking *Reframe*)::
Runs the reframe worker.
The prompt carries the text of *checked* items only -- no unchecked item's text is included.
The result rewrites the intent statement.

*Integrator* (automatic)::
Runs automatically after a debounced human batch (resolving an edit, editing item text, or adding an item), and never while another round is in flight.
Folds your changes back into the space.

== Pending edits (old to new pairs)

When a worker proposes an edit to a *checked* item, the proposal lands as a pending old-to-new pair rendered on that item.
The old text is shown struck through and the new text is shown inserted.

The checked text never changes until you accept or reject the proposal.
Either way, the item's provenance (origin, evidence, notes) is persisted in the space document.

== Research and the dossier

You can trigger research:

* *Per item* -- a research control on each item.
  The topic is derived from the item's text (slugified: lowercase, spaces to hyphens, non-alphanumeric stripped).
* *Free subject* -- the research input field accepts a free-form subject string.

The research worker is *dossier-first*: before running any live search, it reads `research/{topic}.md` under the space's namespace directory.
When the dossier file exists, its markdown is included verbatim in the round's prompt -- later sessions and later TEPs of the same space reuse the accumulated research automatically.

Research findings land only as:

* Unchecked proposed items (never checked -- the gate enforces this).
* Evidence chips attached to items, each carrying `method`, `checkedAt` (ISO date), and a `dossierRef` pointing at the dossier file the round wrote.

The research worker's query options carry:

* `mcpTools`: `tk-package-version`, `web-fetch`, `repo-explorer` -- live registry checks, doc fetches, and the repo corpus.
* `corpusPaths`: the board corpus under the namespace (TEPs, retros, defect log).

The research gate rejects any action that would check an item or write outside the dossier.

=== The dossier persists at the space level

A research run writes `research/{topic}.md` under the thinking space's namespace in the board sidecar.
Because the board sidecar is committed like every other board artifact, research survives sessions and syncs across machines.
A second research trigger on the same topic -- in any session, for any TEP of that space -- consults the existing dossier before going live again.

== The command field

The command input field accepts plain-language commands that drive the form.
Examples:

* `"add this constraint: the API must be versioned"` -- adds an item to the constraints section.
* `"accept all constraints"` -- checks every unchecked active item in constraints.
* `"accept all lists"` -- checks every unchecked active item in every section.
* `"defer the third gap item"` -- marks an item deferred.
* `"remove the optional element about caching"` -- drops an item.

Every returned action carries `actor:"human"` -- the utterance is your act, the interpreter only translates.
The field disables while interpretation is in flight.
An unrecognized utterance returns an explanation rendered under the field -- no model change.

*Freeze is not in the interpreter's vocabulary.*
A model reply containing a freeze action is caught by the gate inside the interpreter and surfaces as an explanation, never as a model change.

== Coverage and readiness

Coverage is item-derived:

* A non-goal section is *covered* when it has at least one checked, active item.
* The goal section is *covered* when the intent text is non-empty.

The *Check readiness* control runs a non-committing dry run of the slicer against your current intent text and records the result.
Freeze enablement reads the *latest* readiness record.

== Freeze and the multi-TEP life of a space

The *Freeze* button is enabled only when:

1. Every section is covered (all non-goal sections have a checked active item; the goal section has non-empty intent text).
2. The latest readiness check returned a clean cut from the dry-run slicer.

When blocked, the button's `data-reason` attribute names the first failing signal:

* `coverage:<kind>` -- the named section lacks a checked active item (e.g. `coverage:constraints`).
* `dryrun:<kind>` -- the dry run could not cut cleanly, pointing at the offending section.
* (empty string) -- the button is enabled; both signals are green.

The freeze pipeline (on your click):

1. Project the checked, active items (state `active` and checked) into a TEP body, grouped by section.
   A superseding item's line carries `(supersedes <shipped item text>)`.
2. Stamp the body with a provenance signature.
3. Write the TEP artifact with `status: "proposed"`.
4. Mark the projected items as `shipped` with their TEP id.
5. Save.

After a freeze:

* Shipped items render locked with `data-state="shipped"` and `data-shipped-in="<tep>"`.
* The space remains open and editable -- the intent editor stays, new items can be added.
* A later prefill round's prompt includes the shipped items and their evidence dossier references.
* The *next* freeze ships only the newly checked, active items -- shipped items are never re-projected.

=== Deferred items

Mark an item deferred instead of dropped to keep it in the space without checking it.
Deferred items persist with their modality, evaluations, and evidence, and reappear whenever the space is reopened.

=== Superseding a shipped item

A new item can supersede a shipped item by declaring which item it supersedes.
Both items carry visible markers: the new item shows which item it supersedes, the shipped item shows which item supersedes it.

The superseding item ships in the next TEP with a supersedes annotation in the body.

== Configuration

[cols="2,3",options="header"]
|===
| Setting | Description

| `thinkube.thinkingSpace.root`
| Board sidecar root directory.
  Thinking-space documents live at `<root>/<namespace>/thinking/<name>.json`.
  Dossier files live at `<root>/<namespace>/research/<topic>.md`.
  Empty = in-memory only (no files written).

| `thinkube.orchestrator.workerModel`
| Model used by all worker rounds (prefill, reframe, integrator, research, interpreter).
  Default: `sonnet`.
|===
