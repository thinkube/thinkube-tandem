= The Spec lifecycle
:page-type: concept
:page-docref: lc9v

A Spec is the unit that runs the full arc from an idea to a merged change. You define it and its acceptance criteria up front, approve it once, and accept it once at the end; between those two points the pair and the orchestrator carry it through slicing, execution, and verification without pausing for you. This page walks that arc and names who acts at each stage, which gate stands between the stages, and what each gate refuses.

== The arc from idea to merge

The whole path, from the first proposal to the merged PR, looks like this. The pair (you and Claude) authors; you alone approve, orchestrate, and accept; the orchestrator runs the work and the closing gate.

[mermaid]
----
flowchart TD
    A["Pair authors the TEP"] --> B["Pair authors the Spec, acceptance criteria first"]
    B --> V{"Every AC certified verifiable or assessment, each with a verification entry?"}
    V -->|"no — blocks Ready"| B
    V -->|"yes — the Spec is ready for review"| C{"Human approves the Spec in the review panel"}
    C -->|"Approve — mints the approval token slice creation requires"| D["Pair cuts vertical slices, status ready"]
    D --> E["Human runs Orchestrate"]
    E --> F["Orchestrator runs each slice's work units on branch spec/SP-n, in its own worktree"]
    F --> G["Closing gate at quiescence: run every declared AC verification"]
    G --> H{"All green?"}
    H -->|"yes — ACs checked, slices Done, verified commits"| I{"Human accepts"}
    H -->|"still red after self-heal and auto-attend — requires-attention"| J["Attend, then re-run"]
    J --> E
    I -->|"Accept & merge"| K["One PR: spec/SP-n squash-merges to main"]
    I -->|"Request changes"| J
----

== Define the work: a TEP, then a Spec

The arc starts with a TEP (Tandem Enhancement Proposal) — the reason above the work, the goal and the forces recorded before any Spec. The Specs that carry out the work nest beneath it.

A Spec is authored acceptance-criteria-first. You give the user-observable outcomes before any design; a good criterion reads like an outcome you could check, not a step you would take. With the criteria agreed, the pair drafts the constraints, the design, and a file plan against your codebase, then runs a verifiability audit over the criteria. Each criterion is interrogated for whether it can be checked, by which actor, and in which environment, before the gate it guards — and is certified either _verifiable_ (a runnable command proves it) or an _assessment_ (an independent reader judges it). Every certified criterion carries a verification entry recording how it is checked.

That full set of entries is the → Ready gate. A Spec's slices may reach Ready only when every acceptance criterion is certified and carries its entry; an undeclared or unresolved criterion blocks the gate and names the offending ordinal. Nothing runs until this passes.

== Approve the Spec

Authoring the Spec crystallizes the plan; it does not start the build. Before the pair can cut a single slice, you open the review panel and approve the Spec. This is a maintainer-only button: Claude can open the review surface for you, but it cannot approve on your behalf, because approving mints the content-bound token that slice creation and the Spec's move to Ready both verify — without it, cutting a slice is refused. Approval is one of the two points in the whole arc where the human acts on the substance.

== Slice the Spec

With the approval in place, the pair cuts the Spec into slices — each a coherent, vertical, end-to-end change you verify and commit as a single "done," sized by coherence rather than by layer or file. A slice that spans more than one file pins a shared contract so every worker builds to the same seam, and decomposes into work units the orchestrator can schedule. Each slice is written to the board at status `ready`, and every acceptance criterion maps to at least one slice. See xref:parallel-execution.adoc[Work units and parallel execution] for how the units fan out.

== Orchestrated execution

The whole Spec runs on one branch, `spec/SP-n`, in its own worktree so parallel Specs never collide. Every slice lands as commits on that branch — no per-slice branch, no per-slice PR. Provisioning the worktree is automatic: with no declared "Worktree setup" recipe, the runner detects the repo's tracked manifests per directory (`package-lock.json` → `npm ci`, `uv.lock` → `uv sync`, and so on, always lockfile-pinned); a declared recipe overrides the detection.

Within a run, completed work survives setbacks: each landed work unit is committed to the spec branch as a `wip(<unit>)` checkpoint. A reset returns the worktree to those checkpoints, never to zero, and a re-dispatch schedules only the un-checkpointed units plus the units of the role the judge found at fault — which rework from their checkpoint, not from scratch. A slice that returns with every unit checkpointed is graded as-is, with no workers dispatched.

Once you run Orchestrate, the slices execute consecutively and autonomously. You defined the goal up front; the orchestrator now runs the Ready slices through to completion without pausing between them and without asking you to sign off on each one. There is no mid-run bless point — the per-slice verification stays automated, and the single human gate waits at the end.

== The closing gate

When every work unit has landed — Spec quiescence — the orchestrator does not silently mark the Spec Done. It runs the acceptance criteria's declared verifications as one complete plan, in dependency order, and attributes each result back to the criterion it proves. The outcome gates both Done and the commit:

* *All green* — it checks exactly the satisfied criteria on the Spec, advances the slices to Done, and lands the verified slice commits on `spec/SP-n` on top of the per-unit checkpoints.
* *Any red or un-runnable* — it works its automated remedies first. A probe that cannot execute is a gate defect, self-healed by re-authoring and re-signing the verification via the auditor (retried once, no rework attempt burned, no role blamed). A failure identical to the prior attempt trips the circuit breaker and escalates immediately. A genuine red goes through bounded judge-routed rework, and an escalated slice gets one automated auto-attend fix. Only a still-red slice leaves the Spec _requires-attention_ for a human. Landed units stay checkpointed on the branch; nothing merges, and the failure stays auditable.

There is no skip. A Spec whose criteria cannot all be verified is never advanced to Done on an un-run verification.

== The delivery report

On every completion, pass or fail, the gate writes a durable report to the Spec's `DELIVERY.md`. It is the operator's document, written plain-language first: a one-line orchestration summary, then _what happened_ (on a failure, the closing gate's diagnosis in full), the acceptance criteria each with its verdict, any out-of-scope findings the workers reported, the touched files, your available exits, and a demoted evidence appendix holding the raw per-criterion runner output. The report opens rendered in the Markdown preview when the run finishes, so you read the outcome before you choose an exit.

== Acceptance: the one human bless point

Acceptance is the single point where you sign off on the assembled result — the one human bless point in the arc. The exits you are offered are derived from the run's terminal state, not a fixed pair glued onto every outcome.

A *delivered* Spec — committed, gate green — offers two:

* *Accept & merge* performs the gated squash-merge of `spec/SP-n` into `main` — `main` gets one verified commit per Spec, and the working trail stays on the PR. It is refused unless every criterion is checked; there is no accepting an unverified Spec.
* *Request changes* opens a primed session to steer the delivered change back toward the intent.

A *stalled* run — the gate did not pass; landed units stay checkpointed on the branch, and nothing merges — offers the actions that apply to a run that did not deliver:

* *Attend* opens a primed fix session in the canonical repo on the requires-attention slice; the fix is applied in the Spec's worktree (named in the prompt) and committed to the spec branch. By the time it is offered, the loop has already run judge-routed rework, the gate self-heal, the circuit breaker, and one automated auto-attend fix — what remains is usually a call only a human can make.
* *Re-run* re-runs the orchestrator once the requires-attention slice is resolved. An attend-fixed slice, fully checkpointed, is graded as-is with no workers dispatched.

The exit states and their actions map out like this:

[mermaid]
----
stateDiagram-v2
    [*] --> Quiescence: all work units landed
    Quiescence --> Delivered: all green — ACs checked, slices Done, verified commits
    Quiescence --> Stalled: still red after self-heal and auto-attend — nothing merges
    Delivered --> Merged: Accept & merge (refused unless every AC checked)
    Delivered --> Quiescence: Request changes — primed rework session, then re-run
    Stalled --> Quiescence: Attend, then Re-run
    Merged --> [*]
----

== One PR per Spec

A Spec produces exactly one PR. Every slice's commits — unit checkpoints, verified slice commits, and any attend fixes — sit on the one `spec/SP-n` branch, and acceptance *squash-merges* that branch to `main`: one verified commit per Spec on `main`, with the full working trail preserved on the PR.

== When a Spec goes stale

A Done slice can go stale after the fact when its parent Spec changes. Only substantive edits count: changes to the acceptance criteria text, the design, or the constraints mark the Spec's Done slices stale for re-verification, because they change what "correct" means. Metadata edits — column moves, theme, priority, due date — and ticking an acceptance checkbox never mark a slice stale, because they record completion or bookkeeping, not a changed requirement. Staleness is measured from a hash of the Spec's requirement sections with the checkbox state stripped, so ticking a box can never trip it. A re-run re-verifies the stale slices against the current Spec before the Spec can reach Done again.
