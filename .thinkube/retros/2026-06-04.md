## 2026-06-04 14:46

### learned

A board-writing extension change can't be dogfooded by the session that writes it. When we moved SP-2's own slices to Done, the running MCP server (and the webview) kept the pre-change build, so the new provenance-stamping code never ran — `verified_req_hash` got stamped but `commit`/`pr` didn't. The new `dist/` only takes effect after a window reload. Implication: to dogfood a change to the MCP server or kanban webview, reload first, then exercise it.

## 2026-06-04 14:46

### learned

"Captured-at-Done" provenance records the branch HEAD commit + the _open_ PR at the instant of the move — not the eventual squash-merge SHA. This surfaced a bootstrap paradox: SP-2's own three slices were marked Done before the delivering commit/PR existed, and PR #14 is now merged/closed, so the feature can't backfill its own provenance. Only Specs delivered _after_ SP-2 get clean capture — and only when the slice is moved to Done while its work sits on a branch with an open PR.

## 2026-06-04 17:44

### learned

Git worktree discovery has a `.git` file-vs-dir asymmetry that split our two board surfaces. A linked worktree's `.git` is a FILE (a gitdir pointer), not a directory — so the navigator's `discoverRepos` (which matched only a `.git` *directory*) didn't detect worktrees at all, while the MCP `walkForBoards` (which keys off `.thinkube/` presence) registered them as rogue standalone boards named by bare dir basename. The surfaces disagreed. Fix (SP-5_SL-4): detect `.git`-file worktrees in both, parse the gitdir pointer (`gitdir: <canonical>/.git/worktrees/<name>`) for the canonical repo, and label them as a worktree of that repo.

## 2026-06-04 17:44

### learned

Monotonic number allocators break across git worktrees — the user caught this mid-spec. `nextSpecNumber()` = max(existing SP-{n}) + 1 is only correct against a single current view of *all* numbers; two divergent worktree checkouts can each compute the same max+1 and mint a duplicate Spec number that collides on merge. Resolution (SP-5_SL-3): canonical-only minting — Spec numbers are allocated only against the canonical repo (resolved via `git rev-parse --git-common-dir` / `git worktree list` first entry), and worktrees only ever *work* an already-numbered Spec. Slice numbers stay safe by single-owner-per-Spec (one worktree per Spec).

## 2026-06-04 17:44

### learned

The parallel-Spec commit tangle bit during SP-5's *own* merge — a live proof of the exact problem SP-5 solves. SP-4's uncommitted `create_slice` work and SP-5's worktree-discovery change both lived in `src/mcp/kanbanMcpServer.ts`, so a plain `git add` of that file would have swept SP-4 into SP-5's PR. Had to partial-stage only SP-5's two hunks (sub-patch built by dropping later hunks, applied with `git apply --cached`) and verify the SP-5-only tree typechecks + passes tests independently (stash the SP-4 delta, compile, pop) before merging. Once worktrees are adopted, the two Specs would have been in separate trees from the start and none of this surgery would be needed.
